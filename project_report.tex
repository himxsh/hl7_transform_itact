% ABSTRACT
%---------------------------------------------------------------------
\begin{center}
    \textbf{Abstract}
\end{center}

This report presents a secure, legal-compliant orchestration pipeline for Health Level
Seven (HL7) clinical messages, specifically addressing large-scale data ingestion from
the MIMIC-IV v2.2 clinical dataset. The system wraps the open-source
\texttt{hl7\_transform} library with novel engineering contributions: a memory-efficient
chunked preprocessing engine capable of streaming 2.4+ GB compressed CSV files, a
deterministic pseudonymization layer using seeded \texttt{Faker} instances for
DPDP Act 2023 compliance, and a cryptographic SHA-256 message-signing module that
appends tamper-evident \texttt{ZSH} Z-segments for IT Act 2000 §43A adherence.
A standalone integrity verifier provides post-hoc auditing of all generated HL7 files.
The resulting pipeline supports both MIMIC-IV and generic CSV datasets, and is
visualized through an interactive React dashboard featuring real-time Server-Sent Event
(SSE) streaming, encryption comparison analytics, and compliance status reporting.
The system demonstrates end-to-end auditable, privacy-preserving, and legally compliant
clinical data processing suited for regulated Indian healthcare environments.

\section{Introduction}

Healthcare interoperability relies heavily on the HL7 (Health Level Seven) messaging
standard for exchanging clinical, administrative, and financial data between hospital
information systems. However, integrating massive clinical research datasets such as
MIMIC-IV into modern research environments while strictly adhering to Indian data
protection laws demands a purpose-built architecture that goes beyond simple format
conversion. Raw clinical data contains Personally Identifiable Information (PII) and
Protected Health Information (PHI) that must be systematically scrubbed, and the
resulting messages must carry tamper-evident seals to satisfy regulatory security
requirements.

This project addresses these challenges by constructing a modular orchestration pipeline
that transforms structured clinical CSV data into standardized, privacy-scrubbed, and
cryptographically verifiable HL7 \texttt{ORU\string^R01} messages. The pipeline is
built atop the open-source \texttt{hl7\_transform} library, extending it with custom
preprocessing, anonymization, integrity signing, and a full-stack React dashboard for
operational monitoring.

\subsection*{Problem Statement}

When utilizing clinical datasets for research or transmitting live hospital data,
organizations face three simultaneous obligations: (1) preserve the structural integrity
and clinical utility of the data, (2) protect patient privacy by removing or masking all
PII and PHI, and (3) ensure that data has not been tampered with during storage or
transit. Existing HL7 transformation tools handle format conversion but provide no
built-in compliance layer. The core challenge is to build an orchestration pipeline that
translates raw CSV or generic HL7 feeds into standardized, privacy-scrubbed, and
cryptographically verifiable HL7 messages --- all while operating within the legal
frameworks of the IT Act 2000 and the DPDP Act 2023.

\subsection*{Novelty and Contribution}

\begin{itemize}
    \item \textbf{High-Efficiency Chunked Ingestion}: A memory-efficient preprocessing
    engine tailored for MIMIC-IV v2.2, capable of streaming the 2.4+ GB compressed
    \texttt{labevents.csv.gz} file in configurable chunks (default: 100,000 rows per
    chunk), performing multi-table relational joins on \texttt{subject\_id}, and mapping
    structured CSV data into valid HL7 \texttt{ORU\string^R01} messages without memory
    overflow.

    \item \textbf{DPDP-Aligned Deterministic Privacy Engine}: An \texttt{Anonymizer}
    class utilizing a seeded \texttt{Faker} instance with \texttt{en\_IN} locale. By
    seeding the random generator with the unique \texttt{subject\_id}, the system
    ensures that a patient appearing in multiple lab events is consistently masked to
    the same pseudonymous Indian identity (name, address, demographics), preserving
    longitudinal relational utility while fully scrubbing real PII. Additionally,
    compiled regex patterns detect and redact SSNs, Aadhaar numbers, PAN cards,
    emails, phone numbers, and medical record numbers from clinical free-text notes.

    \item \textbf{IT Act 2000 Integrity Protocol}: The \texttt{IntegrityManager}
    computes a SHA-256 hash of the entire canonical HL7 message payload and appends
    it as a custom \texttt{ZSH} Z-segment with five fields: set ID, algorithm name,
    hex digest, signing status, and ISO 8601 timestamp. A standalone
    \texttt{validate\_integrity.py} script provides independent post-hoc verification
    by stripping the ZSH segment, re-hashing the payload, and comparing digests.

    \item \textbf{Dual-Pipeline Architecture}: The orchestrator (\texttt{main.py})
    supports two distinct pipeline modes --- a specialized MIMIC-IV pipeline with
    chunked preprocessing and a generic CSV pipeline with configurable column
    mappings --- allowing the system to handle diverse clinical datasets.

    \item \textbf{Interactive React Dashboard}: A modern front-end built with React,
    TypeScript, and Tailwind CSS that provides real-time SSE streaming of pipeline
    progress, encryption comparison analytics, data lineage visualization, and
    compliance status reporting.
\end{itemize}

\subsection*{Objectives}

\begin{itemize}
    \item Build a modular and extensible HL7 orchestration pipeline that handles both
    MIMIC-IV and generic CSV clinical datasets.
    \item Implement privacy-by-design controls aligned to the DPDP Act 2023, including
    deterministic pseudonymization and PII scrubbing.
    \item Enforce data integrity through cryptographic SHA-256 signing with custom
    Z-segment embedding, satisfying IT Act 2000 §43A requirements.
    \item Provide a standalone integrity verification tool for post-hoc compliance
    auditing.
    \item Deliver an interactive React dashboard that surfaces pipeline metrics,
    security analytics, and compliance evidence for technical and non-technical
    reviewers.
    \item Deliver an academically rigorous, deployment-ready, and legally compliant
    artifact suitable for regulated clinical environments.
\end{itemize}

\subsection*{Scope}

\begin{itemize}
    \item \textbf{Dataset}: MIMIC-IV v2.2 clinical data (\texttt{patients.csv.gz},
    \texttt{labevents.csv.gz}, \texttt{d\_labitems.csv.gz}) and generic CSV datasets
    (e.g., Indian Liver Patient Dataset).
    \item \textbf{Message Standard}: HL7 v2.x \texttt{ORU\string^R01} observation

    \item \textbf{Privacy Layer}: Deterministic pseudonymization (PID segments) and

    \item \textbf{Security Layer}: SHA-256 cryptographic hashing with ZSH Z-segment

    \item \textbf{Product Layer}: React/TypeScript dashboard with SSE streaming,
    encryption analytics, data lineage, and compliance views.
    \item \textbf{Governance Layer}: Legal mapping to IT Act 2000 (§43A, §72A),
    DPDP Act 2023, and SPDI Rules 2011; risk articulation and audit evidence.
\end{itemize}

\section{Methodology}

The system follows a layered architecture with explicit inputs, outputs, and compliance
checkpoints at each stage. Every transformation is designed to be measurable, auditable,
and independently verifiable.

    \centering
    \includegraphics[width=\textwidth]{diagrams/system_architecture.png}
    \caption{HL7 Orchestration System Architecture}

The technical workflow is divided into five core functional layers: ingestion, privacy, 
security, verification, and monitoring.

\subsection{Data Ingestion and Preprocessing}

\textbf{Input sources.} Three compressed CSV files from MIMIC-IV v2.2:
\texttt{patients.csv.gz} (demographics), \texttt{labevents.csv.gz} (laboratory
observations, 2.4+ GB compressed), and \texttt{d\_labitems.csv.gz} (lab item
dictionary).

\textbf{Column minimization.} Only the columns required for HL7 message construction
are retained at load time:
\begin{itemize}
    \item \texttt{patients}: \texttt{subject\_id}, \texttt{gender},
    \texttt{anchor\_age}, \texttt{anchor\_year}, \texttt{dod}
    \item \texttt{labevents}: \texttt{subject\_id}, \texttt{itemid},
    \texttt{charttime}, \texttt{value}, \texttt{valuenum}, \texttt{valueuom},
    \texttt{ref\_range\_lower}, \texttt{ref\_range\_upper}, \texttt{flag},
    \texttt{comments}
    \item \texttt{d\_labitems}: \texttt{itemid}, \texttt{label}, \texttt{fluid},
    \texttt{category}
\end{itemize}

\textbf{Chunked streaming.} The \texttt{labevents.csv.gz} file is too large to fit
in memory. The pipeline reads it in chunks of 100,000 rows using
\texttt{pandas.read\_csv(chunksize=100000)}, filters each chunk to retain only rows
matching the sampled patient IDs, and concatenates the filtered results. This approach
reduces peak memory usage from several gigabytes to under 500 MB.

\textbf{Relational joins.} The three datasets are merged on \texttt{subject\_id} and
\texttt{itemid} to produce a single enriched DataFrame containing patient demographics,
lab test metadata, and observation values.

\textbf{Storage posture.} All intermediate artifacts remain local. No PII is emitted to
dashboards, logs, or external services. The preprocessing module logs only metadata
(row counts, column names, processing times) for compliance audit purposes.

\subsection{Privacy Preservation (Anonymizer)}

The \texttt{Anonymizer} class is the core DPDP compliance component. It operates on
two orthogonal axes: structured PID segment replacement and unstructured free-text
PII scrubbing.

\textbf{Deterministic pseudonymization.} For a patient $P_i$ with unique identifier
\texttt{subject\_id}:
\begin{equation}
    \text{Pseudonym}(P_i) = \text{Faker}(\text{seed} = P_i.\text{subject\_id})
\end{equation}

By seeding the \texttt{Faker} random generator with the patient's unique ID and using
the \texttt{en\_IN} locale, the system guarantees three properties:
\begin{enumerate}
    \item \textbf{Consistency}: The same patient always maps to the same fake Indian
    name, address, and phone number across all their lab events.
    \item \textbf{Referential integrity}: Longitudinal studies can still link multiple
    records for the same patient without revealing their real identity.
    \item \textbf{No mapping table}: The seed \emph{is} the mapping --- no external
    lookup table needs to be stored or secured.
\end{enumerate}

\textbf{PID segment fields replaced}: Patient name (PID.5), address (PID.11), phone
number (PID.13), and SSN/identifier (PID.19).

\textbf{Free-text PII scrubbing.} The \texttt{scrub\_notes} method applies compiled
regex patterns to NTE (Notes and Comments) segments:

\subsection{Security and Integrity (IT Act)}

Under the IT Act 2000, Section 43A (Reasonable Security Practices) and Section 72A
(Breach of Confidentiality and Privacy), the pipeline enforces data integrity through
cryptographic hashing.

\textbf{Hash computation.} For a completed HL7 message $M$:
\begin{equation}
    \text{Hash}_{msg} = \text{SHA-256}(M_{\text{canonical}})
\end{equation}
where $M_{\text{canonical}}$ is the full HL7 message content excluding the ZSH segment
itself.

\textbf{ZSH segment format.} The hash is packaged into a custom Z-segment with five
fields:
\begin{verbatim}
ZSH|1|SHA256|<hex_digest>|SIGNED|<ISO-8601 timestamp>
\end{verbatim}

\begin{itemize}
    \item \textbf{ZSH.1} (Set ID): Always ``1''
    \item \textbf{ZSH.2} (Algorithm): ``SHA256''
    \item \textbf{ZSH.3} (Digest): 64-character hex-encoded SHA-256 hash
    \item \textbf{ZSH.4} (Status): ``SIGNED''
    \item \textbf{ZSH.5} (Timestamp): UTC signing time in ISO 8601 format
\end{itemize}

The \texttt{IntegrityManager} is stateless --- both \texttt{sign\_message()} and
\texttt{verify\_message()} operate without modifying instance state, ensuring
thread-safety and idempotency.

\subsection{Integrity Verification}

The \texttt{validate\_integrity.py} script provides independent post-hoc verification:
\begin{enumerate}
    \item Parse each \texttt{.hl7} file from the output directory.
    \item Locate and extract the trailing \texttt{ZSH} segment.
    \item Strip the \texttt{ZSH} line from the message body.
    \item Recompute \texttt{SHA-256} over the remaining canonical content.
    \item Compare the recomputed digest against \texttt{ZSH.3}.
    \item Report PASS/FAIL with detailed logging to \texttt{pipeline.log}.
\end{enumerate}

The script supports batch validation (entire output directory) and single-file
validation, returning exit code 0 for all-pass and exit code 1 if any file fails.

    \State \textbf{Visualize}: Push metrics to React Dashboard via SSE
\end{algorithmic}

\section{Operational Working}

This section describes how the end-to-end system is exercised in practice, what
evidence is produced at each stage, and how the frontend dashboard surfaces operational
and compliance information.

\subsection{Pipeline Execution Flow}

\begin{enumerate}
    \item \textbf{Initialization}: The user launches the orchestrator via
    \texttt{python main.py --type mimic --sample 50} or through the React frontend
    UI, which triggers the FastAPI backend.
    \item \textbf{Data routing}: Based on the \texttt{--type} argument, the pipeline
    routes to either \texttt{mimic\_pipeline.py} (specialized chunked processor) or
    \texttt{generic\_pipeline.py} (configurable column-mapping processor).
    \item \textbf{Preprocessing}: For MIMIC-IV, the preprocessor streams
    \texttt{labevents.csv.gz} in 100K-row chunks, filters to sampled patients,
    performs three-way relational joins, and produces an enriched DataFrame.
    \item \textbf{HL7 construction}: Each row is transformed into a valid
    \texttt{ORU\string^R01} message with MSH, PID, PV1, OBR, OBX, and NTE segments
    using the \texttt{hl7\_transform} mapping engine.
    \item \textbf{Anonymization}: The \texttt{Anonymizer} replaces PID segment fields
    with deterministic pseudonyms and scrubs NTE free-text using compiled regex
    patterns.
    \item \textbf{Cryptographic sealing}: The \texttt{IntegrityManager} computes the
    SHA-256 hash of the canonical message and appends the ZSH Z-segment.
    \item \textbf{File output}: Signed messages are written to
    \texttt{output/<subject\_id>.hl7}, one file per patient.
    \item \textbf{Verification}: \texttt{validate\_integrity.py} is run to re-hash
    every output file and compare against the stored ZSH seal, logging PASS/FAIL
    results.
    \item \textbf{Dashboard presentation}: The React frontend displays real-time
    progress via SSE, then renders security analytics, encryption comparisons, and
    compliance status.
\end{enumerate}

\subsection{Dashboard Pages and Components}

\begin{itemize}
    \item \textbf{Landing Page}: Project overview with architecture visualization and
    navigation to all pipeline features.
    \item \textbf{Database Selection}: Choose between MIMIC-IV and Generic CSV
    pipelines; configure sample size and output directory.
    \item \textbf{Real-time Dashboard}: Live SSE streaming of pipeline progress ---
    records processed, anonymization status, signing status, with animated progress
    indicators.
    \item \textbf{Security Analysis Modal}: Post-pipeline encryption comparison
    showing before/after anonymization diffs, hash verification results, and ZSH
    segment details for each processed record.
    \item \textbf{Data Lineage Page}: Visualization of the transformation chain from
    raw CSV $\rightarrow$ HL7 $\rightarrow$ anonymized HL7 $\rightarrow$ signed HL7,
    showing exactly what was modified at each stage.
    \item \textbf{Architecture Page}: Technical documentation of system components,
    data flows, and compliance mappings rendered as interactive UI elements.
    \item \textbf{Audit Log}: Searchable, filterable view of all pipeline operations
    with timestamps, record counts, and processing metadata.
\end{itemize}

\subsection{Governance, IT Act, and Data Protection Alignment}

The project embeds accountability principles from Indian IT law, the DPDP Act 2023,
and the SPDI Rules 2011 directly into the pipeline's operational flow.

\subsubsection{IT Act 2000: Sections 43A and 72A}

\begin{itemize}
    \item \textbf{Reasonable security practices (Sec. 43A)}: The SHA-256 cryptographic
    hashing with ZSH Z-segment embedding provides message-level tamper evidence.
    The independent \texttt{validate\_integrity.py} verifier demonstrates that
    reasonable security practices are not only implemented but independently
    auditable. Column minimization during preprocessing and deterministic
    pseudonymization further establish a defense-in-depth security posture.

    \item \textbf{Breach of confidentiality (Sec. 72A)}: The deterministic anonymizer
    ensures that no real patient identity is present in any output file. The audit
    log records all pipeline operations with metadata (timestamps, record counts,
    processing modes) without storing any PHI. If a breach investigation is required,
    the audit trail provides a complete chronology of data access and transformation
    events.
\end{itemize}

\subsubsection{Digital Personal Data Protection Act 2023}

\begin{itemize}
    \item \textbf{Data minimization (§8)}: The preprocessing engine retains only the
    columns strictly necessary for HL7 message construction. All other fields from
    the MIMIC-IV source files are dropped at load time, reducing the data surface
    area.

    \item \textbf{Purpose limitation (§8(4))}: The anonymizer documentation explicitly
    references DPDP §8(4), stating that only data fields required for clinical lab
    reporting are retained. The pipeline enforces this by construction --- the column
    lists are hardcoded constants.

    \item \textbf{De-identification (§8(7))}: The \texttt{Anonymizer} class implements
    reasonable de-identification as mandated by §8(7). The seeded Faker approach
    provides stronger guarantees than simple masking because it generates
    structurally valid replacement identities that preserve data utility for
    downstream analysis.

    \item \textbf{Storage limitation}: All intermediate and output artifacts remain
    local to the pipeline directory. The system does not transmit any data to
    external services. Retention policies can be applied by periodically purging
    the \texttt{output/} directory.

    \item \textbf{Security safeguards}: The combination of column minimization,
    deterministic pseudonymization, cryptographic hashing, and independent
    verification provides a multi-layered security posture aligned with DPDP
    expectations.
\end{itemize}

\subsubsection{SPDI Rules 2011 and International References}

\begin{itemize}
    \item \textbf{SPDI Rules 2011}: The Sensitive Personal Data or Information Rules
    require organizations to implement ``reasonable security practices and
    procedures.'' The SHA-256 signing and independent verification satisfy this
    requirement at the message level.

    \item \textbf{HIPAA alignment}: While not directly applicable in India, the
    pipeline's approach to de-identification (Safe Harbor method analog via
    systematic PII scrubbing) and audit logging aligns with HIPAA Privacy Rule
    best practices.

    \item \textbf{GDPR Art. 5(2) accountability}: The audit log and independent
    verification script provide the evidence trail that GDPR's accountability
    principle demands, should the system be deployed in a cross-border context.
\end{itemize}

\section{Results}

\subsection{Functional Outcomes}

The delivered system successfully supports:
\begin{itemize}
    \item End-to-end HL7 message generation from raw MIMIC-IV compressed CSV files
    with configurable sample sizes (tested with $N = 50$, producing 70+ lab event
    HL7 messages due to one-to-many patient-to-event mapping).
    \item Generic CSV pipeline execution with configurable column mappings (tested

\section{Conclusion}

This project demonstrates that healthcare data interoperability and strict legal
compliance are not mutually exclusive objectives. By wrapping the open-source
\texttt{hl7\_transform} library with purpose-built modules for chunked MIMIC-IV
ingestion, deterministic DPDP-aligned pseudonymization, and IT Act-compliant
cryptographic integrity sealing, the resulting orchestration pipeline provides a
comprehensive, scalable, and legally defensible solution for clinical data processing.

The key technical contributions --- memory-efficient streaming of 2.4+ GB datasets,
seeded Faker-based anonymization preserving relational integrity without mapping
tables, SHA-256 ZSH Z-segment signing with independent verification, and a dual
MIMIC/Generic pipeline architecture --- collectively demonstrate that compliance can
be embedded into the data processing pathway rather than bolted on as an afterthought.

The React dashboard ensures that these complex privacy and security operations are
transparent, auditable, and accessible to both technical implementers and compliance
officers. Future extensions include migration to FHIR R4 message formats, integration
of quantum-resistant hash algorithms, expansion of the regex PII pattern library for
emerging Indian ID formats, and deployment of the pipeline as a containerized
microservice for production hospital environments.
